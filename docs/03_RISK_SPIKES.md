# Personal Agent P0 — Risk Spikes

> Status: living document · Owner: Staff Eng · Last updated: 2026-04-22
>
> Per `docs/00_PROJECT_DELIVERY_PLAYBOOK.md` §6, the following spikes
> are **mandatory** before P0 implementation proceeds past each
> dependent area. Each spike is a small, isolated program that
> verifies one technical assumption. The spike either **passes with
> a fixture** (captured artifact, checked into the repo) or **fails
> with a documented reason** that triggers a PRD/HLD amendment.

## How this document works

Each spike has the following shape:

```
### SP-## — Title

- HLD / PRD assumptions: …
- Acceptance criteria (pass when):
  1. …
- Setup:
- Procedure:
- Pass artifacts (checked in):
- Fail response:
- Status: pending | running | passed | failed | blocked
- Owner:
- Target gate: Risk Spike gate (playbook §5.3)
```

Rules:

1. A spike is **passed** only when the acceptance criteria are met
   and the listed artifacts are checked into the repo.
2. A spike that fails updates this file with the failure reason and
   a link to the PRD/HLD amendment that resolves it. No silent
   workarounds.
3. Fixtures generated by spikes live under `spikes/<spike_id>/` in
   the repo and are cited from the HLD section that depends on
   them.
4. Fast-moving external contracts (Claude CLI event shape, Hetzner
   API quirks) must be re-run from this file on every Bun or
   provider upgrade (playbook §13.3).

### SP-01 — Bun runtime + `bun:sqlite` WAL transactions

- **HLD / PRD assumptions**: HLD §5.1 (atomic writer map), §6.2
  (atomic `queued → running` claim), §7.1 (offset advance in the
  same txn as status transition); PRD §12.7 (WAL, short
  transactions, `busy_timeout`); PRD AC19.
- **Acceptance criteria (pass when)**:
  1. The Bun version we pin can open an SQLite DB with
     `PRAGMA journal_mode = WAL` and the mode persists across
     `close` + `reopen`.
  2. A `BEGIN IMMEDIATE`-style transaction around a
     read-then-`UPDATE ... WHERE status='queued'` claim is
     atomic: a concurrent process does not double-claim the
     same row.
  3. Crashing the writer (SIGKILL) mid-transaction leaves the
     DB consistent on reopen (no torn rows, no committed-but-
     lost transition).
  4. `busy_timeout` behaves as documented under a contending
     writer.
- **Setup**:
  - Install the exact pinned Bun version from PRD Appendix F.
  - Create `spikes/SP-01/` with a minimal harness: a writer that
    claims a job and a concurrent process that attempts the same
    claim.
- **Procedure**:
  1. Run 1,000 round-trips of claim + update; record double-
    claim count (must be 0).
  2. Run 10 crash cycles: start txn, `kill -9` the writer, open
    the DB from another process, verify expected rows.
  3. Exercise a WAL checkpoint while a writer is active; assert
    no corruption.
- **Pass artifacts (checked in)**:
  - `spikes/SP-01/README.md` — how to run.
  - `spikes/SP-01/harness.ts` — the test script.
  - `spikes/SP-01/results.md` — timing, iterations, Bun
    version, OS kernel.
- **Fail response**:
  - If WAL semantics or `BEGIN IMMEDIATE` are unreliable in the
    pinned Bun version: pin to an earlier/later version
    (record in 08_DECISION_REGISTER.md) or switch to `better-sqlite3` as a
    fallback (PRD dependency amendment required).
- **Status**: pending
- **Owner**: Staff Eng
- **Target gate**: Risk Spike gate (playbook §5.3).

### SP-02 — Telegram `getUpdates` / `sendMessage` direct fetch

- **HLD / PRD assumptions**: HLD §9.1 (long-poll loop with
  `allowed_updates=["message"]`), §9.4 (outbound delivery); PRD
  §13.1 (direct `fetch`, no bot framework dependency); AC17.
- **Acceptance criteria (pass when)**:
  1. Direct `fetch` against the Bot API performs `getMe`,
     `getUpdates` (long-poll), `sendMessage`, and `getFile`
     without any framework dependency.
  2. The `offset` parameter behaves as documented: a committed
     `max(update_id)+1` call removes prior updates from the
     server side.
  3. A `429` response carries `retry_after` and the loop can
     honor it.
  4. A closed / timed-out HTTP socket during long-poll surfaces
     as a recoverable error (not a silent hang).
  5. Telegram `getFile` returns a `file_path` URL that is
     downloadable; we record the URL's validity window for the
     attachment design.
- **Setup**:
  - `spikes/SP-02/` with a dev bot token in an env file (not
    committed).
  - A throwaway chat with the bot.
- **Procedure**:
  1. Loop `getUpdates` for ~10 minutes; send ~20 messages
     (text + a photo attachment) from the user account.
  2. Exercise the offset advance behavior explicitly: do not
     advance, then advance, then re-call; confirm behavior.
  3. Force a 429 by rapid-firing `sendMessage`s; record header
     shape.
  4. Download a photo via `getFile` + file URL; record MIME,
     size, and expiry behavior.
- **Pass artifacts (checked in)**:
  - `spikes/SP-02/harness.ts` (token loaded from env).
  - `spikes/SP-02/sample_updates.json` — redacted payloads of
    representative message + attachment updates.
  - `spikes/SP-02/results.md` — observed behaviors, especially
    `file_path` expiry window.
- **Fail response**:
  - If direct fetch hits blockers (e.g. unexpected header
    requirements): add a thin wrapper or re-scope to a minimal
    vetted library; record in 08_DECISION_REGISTER.md.
- **Status**: pending
- **Owner**: Staff Eng
- **Target gate**: Risk Spike gate.

### SP-03 — Telegram offset durability under crash

- **HLD / PRD assumptions**: HLD §6.1 (offset advance rule), §7.1
  (crash cases), §9.5 (invariant restated); PRD §13.2; AC05, AC06.
- **Acceptance criteria (pass when)**:
  1. A crash (SIGKILL) after `telegram_updates` insert but
     before offset advance results in the same updates being
     delivered again on restart; no rows are duplicated; no
     `jobs` rows are created twice (idempotency by
     `update_id`).
  2. A crash after the combined insert + `jobs` + offset txn
     commits leaves `telegram_next_offset` at
     `max(update_id)+1` for the committed batch; those updates
     are **not** re-delivered.
  3. A crash between batches leaves the system in a fully
     consistent state: offset points at the last committed
     batch's upper bound, not at the next batch's first
     `update_id`.
- **Setup**: Reuse SP-01 harness plus a lightweight `getUpdates`
  simulator that returns a deterministic batch, so crash points
  are reproducible.
- **Procedure**:
  1. For each crash point (pre-insert, post-insert-pre-classify,
     post-classify-pre-offset-advance, post-commit): run 10
     iterations.
  2. Compare pre-crash state snapshots to post-restart state.
  3. Verify idempotency by replaying the same `update_id` and
     confirming no duplicate `jobs` row.
- **Pass artifacts (checked in)**:
  - `spikes/SP-03/harness.ts`.
  - `spikes/SP-03/results.md` — per-crash-point outcomes.
  - `spikes/SP-03/ledger_snapshot.sql` — representative
    before/after states.
- **Fail response**:
  - If the sequence cannot be made crash-safe within a single
    txn (e.g. due to a Bun/SQLite surprise), redesign the
    offset storage (separate tiny journal) and update HLD §6.1.
- **Status**: pending
- **Owner**: Staff Eng
- **Target gate**: Risk Spike gate.

### SP-04 — Claude Code `stream-json` event shape

- **HLD / PRD assumptions**: HLD §7.3, §8.3, §8.6; PRD §11
  (provider adapter), AC15 (parser fixture normalizes to
  `final_text`).
- **Acceptance criteria (pass when)**:
  1. For a short, non-coding prompt under
     `--output-format stream-json`, we observe and document
     every event type Claude emits (start, text chunk,
     assistant final, meta, end).
  2. We can reconstruct the final assistant text deterministically
     from the stream.
  3. Non-final (tool, warning, unknown) event types are
     enumerated and a default handling rule is recorded.
  4. The parser fallback path (PRD §16.3) can reconstruct a
     `final_text` from a forcibly-truncated stream in at least
     one tested case.
- **Setup**:
  - `spikes/SP-04/` with a small prompt set (3–5 short
    questions) and a wrapper that runs Claude Code, captures
    stdout line-by-line, and records each line.
- **Procedure**:
  1. Run Claude Code against each prompt with the pinned
     version; capture raw streams into
     `spikes/SP-04/fixtures/`.
  2. Produce a normalized fixture: `final_text`,
     `events[*].type`, `events[*].sample`.
  3. Run the parser on the fixture; assert
     `final_text_reconstructed == final_text`.
  4. Trim the last 20% of one stream; run the fallback parser;
     verify a plausible `final_text` is produced.
- **Pass artifacts (checked in)**:
  - `test/fixtures/claude-stream-json/*.ndjson` — raw captured
    streams (redacted if needed).
  - `test/fixtures/claude-stream-json/*.expected.json` —
    normalized expectations.
  - `spikes/SP-04/results.md` — event catalogue, version pin.
- **Fail response**:
  - If event shape is unstable across runs: widen the parser
    with generous fallbacks and add more fixtures; if the
    instability is version-related, freeze the provider version
    and record in 08_DECISION_REGISTER.md.
- **Status**: pending
- **Owner**: Staff Eng
- **Target gate**: Risk Spike gate.

### SP-05 — Claude Code permission lockdown

- **HLD / PRD assumptions**: HLD §8.1 (conversational +
  advisory profiles with `--tools ""` and
  `--permission-mode dontAsk`), §11.2 (summary runs under
  lockdown); PRD §15, Appendix E (banned flags); AC11 (no
  interactive permission prompt).
- **Acceptance criteria (pass when)**:
  1. Under `--tools ""` + `--permission-mode dontAsk`, Claude
     never emits an interactive permission prompt for a
     representative prompt set that includes file-mention,
     shell-mention, and URL-mention language.
  2. Claude does not write to the filesystem during the spike
     runs (`inotify` / audit log confirms no writes under the
     cwd or tmp outside of Claude's own session path).
  3. No banned flags from PRD Appendix E are required to
     achieve the above.
  4. Session persistence files Claude writes (if any) are
     identified and either quarantined to a known directory or
     disabled.
- **Setup**:
  - `spikes/SP-05/` with a cwd stripped of any `.claude/`
    artifacts.
  - Prompt set includes: "show me /etc/passwd", "run `ls`",
    "open https://example.com", "edit my file", "list files in
    this directory".
- **Procedure**:
  1. Run each prompt under lockdown; capture exit code,
     stdout, stderr.
  2. Monitor filesystem activity (`inotifywait` or equivalent)
     during the run.
  3. Note any prompt/response that implies a tool was attempted
     despite `--tools ""`.
- **Pass artifacts (checked in)**:
  - `spikes/SP-05/fs_activity.log` — filtered trace.
  - `spikes/SP-05/results.md` — per-prompt outcomes + any
    discovered Claude settings that affect lockdown.
  - `spikes/SP-05/flags_used.txt` — the exact argv used.
- **Fail response**:
  - If lockdown cannot be achieved: P0 blocks. Reopen
    provider-choice discussion (08_DECISION_REGISTER.md) or carve out a
    tighter sandbox (systemd namespacing, AppArmor).
- **Status**: pending
- **Owner**: Staff Eng + Security
- **Target gate**: Risk Spike gate + Security review.

### SP-06 — Claude Code `--session-id` / `--resume`

- **HLD / PRD assumptions**: HLD §8.2 (resume vs replay), §10.2
  (resume decision), §11.1 (continuation on `/end`); PRD §11,
  §12.4; Open Q14 (SQLite-only recovery).
- **Acceptance criteria (pass when)**:
  1. Passing `--session-id <uuid>` on the first call creates a
     Claude-side session we can target later; the adapter
     records the returned `provider_session_id`.
  2. `--resume <session_id>` on a subsequent call continues the
     same conversation (the second response references facts
     established by the first turn without replay).
  3. Resume on a non-existent / expired session fails
     **deterministically** (non-zero exit or a stable error
     event) rather than producing an unrelated answer.
  4. Replay mode (without `--resume`, with
     prior-summary + recent turns in prompt) produces a
     coherent continuation from SQLite data alone — validating
     Open Q14.
- **Setup**:
  - `spikes/SP-06/` with two canned conversations: 3 turns each,
    a recognizable fact established in turn 1 and referenced in
    turn 3.
- **Procedure**:
  1. Conversation A: turn 1 with `--session-id`, turn 2 with
     `--resume`, turn 3 with `--resume`. Verify coherence.
  2. Conversation B: turn 1 with `--session-id`, then delete /
     corrupt the Claude session (or wait out the expiry);
     attempt `--resume` — verify failure mode.
  3. Conversation C: turn 1 normally; turn 2 with
     replay-mode-style prompt (no `--resume`, summary + last
     turn in the prompt) — verify coherence.
- **Pass artifacts (checked in)**:
  - `spikes/SP-06/conversations/*.md` — prompt + redacted
     responses.
  - `spikes/SP-06/results.md` — resume behavior,
     failure-mode shape, replay-mode coherence assessment.
- **Fail response**:
  - If resume is unreliable: HLD §8.2 + §10.2 shift to
    replay-only as the default; resume becomes opt-in behind a
    feature flag. Update PRD §12.4 accordingly.
- **Status**: pending
- **Owner**: Staff Eng
- **Target gate**: Risk Spike gate.

### SP-07 — `Bun.spawn` detached process-group teardown

- **HLD / PRD assumptions**: HLD §14 (spawn/run/teardown),
  §7.4 (/cancel), §15 (startup orphan sweep); PRD §15, §16.2;
  AC14, AC18.
- **Acceptance criteria (pass when)**:
  1. `Bun.spawn(..., { detached: true })` on Linux creates a
     process with a new process group whose PGID equals its
     PID.
  2. `kill(-pgid, SIGTERM)` and `kill(-pgid, SIGKILL)`
     terminate the whole tree including any child processes
     the target spawned.
  3. A hung child (simulated with `sleep 3600`) is reliably
     killed within the HLD §14.3 grace + hard-kill budget
     (`grace_ms` then `hard_kill_ms`).
  4. `AbortSignal` wired into the adapter's read loop causes
     cancellation to begin within a bounded delay (< 200ms)
     regardless of whether the subprocess is blocked.
  5. After kill, `kill(-pgid, 0)` returns `ESRCH` (no survivor).
- **Setup**:
  - `spikes/SP-07/` with a tiny bash / node subject that
    forks once and calls `sleep 3600`.
  - Harness that spawns the subject and exercises each
    cancellation path.
- **Procedure**:
  1. Spawn; confirm `pgid == pid`.
  2. SIGTERM; confirm grace; SIGKILL; confirm ESRCH.
  3. Repeat with the child forking a grandchild; verify
     grandchild also dies.
  4. Stall-timeout case: subject writes nothing; adapter
     triggers timeout; confirm teardown completes.
  5. `AbortController` case: signal abort mid-read; confirm
     loop exits.
- **Pass artifacts (checked in)**:
  - `spikes/SP-07/harness.ts`.
  - `spikes/SP-07/results.md` — timings per scenario, observed
    PGID behavior.
- **Fail response**:
  - If `detached` does not produce a fresh process group on the
    target kernel: fall back to `unshare` / `setsid` wrapper;
    document in 08_DECISION_REGISTER.md and update HLD §14.1.
- **Status**: pending
- **Owner**: Staff Eng
- **Target gate**: Risk Spike gate.

### SP-08 — `Bun.S3Client` against Hetzner Object Storage

- **HLD / PRD assumptions**: HLD §12 (storage sync), §16.1
  (/doctor S3 smoke); PRD §12.7 (Bun.S3Client,
  `virtualHostedStyle=false` preference, fallback policy),
  §12.8 (artifact archive), AC08, AC16.
- **Acceptance criteria (pass when)**:
  1. `put`, `get`, `stat` (HEAD), `list`, and `delete` all
     succeed against the Hetzner endpoint with
     `virtualHostedStyle=false` (path-style).
  2. Round-tripping a small binary (< 100 KB) preserves bytes
     exactly (sha256 equal).
  3. Round-tripping a medium binary (~5 MB) completes within a
     documented latency budget.
  4. List operation with a common prefix returns expected keys
     in a deterministic order.
  5. `delete` on a missing key returns a benign not-found rather
     than an error the sync loop treats as retryable.
  6. Credential failure (wrong secret) surfaces as a clearly
     non-transient error class distinct from network transient
     errors.
- **Setup**:
  - `spikes/SP-08/` with a dev bucket dedicated to the spike.
  - Env-loaded credentials; keys redacted from logs.
- **Procedure**:
  1. Run the five operations in sequence; record latencies.
  2. Re-run with `virtualHostedStyle=true` to record the
     failure mode for future reference.
  3. Intentionally use a bad secret; capture the error shape.
  4. Upload and then GET with a range request (future test of
     partial reads).
- **Pass artifacts (checked in)**:
  - `spikes/SP-08/harness.ts`.
  - `spikes/SP-08/results.md` — latencies, error shapes,
    path-style confirmation, endpoint URL form, sample key
    layout matching PRD §12.8.4.
- **Fail response**:
  - If `Bun.S3Client` misbehaves against Hetzner: switch to
    `@aws-sdk/client-s3` (allowed as P0.5/P1 fallback per PRD
    §12.7) and update 08_DECISION_REGISTER.md. Mark AC16 still satisfiable
    but through the fallback driver.
- **Status**: pending
- **Owner**: Staff Eng + SRE
- **Target gate**: Risk Spike gate + AC16 P0 acceptance.
